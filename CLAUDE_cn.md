# CLAUDE_cn.md

减少常见 LLM 编码失误的行为准则。根据项目具体情况合并使用。

**取舍说明：** 这些准则偏向谨慎而非速度。对于简单任务，可酌情判断。

## 1. 先思考，再编码

**不假设，不掩盖困惑，呈现权衡。**

实现前：
- 明确陈述你的假设。如有不确定，主动提问。
- 若存在多种解读，逐一列出——不要暗自选择其中一种。
- 若有更简单的方案，说出来。必要时提出异议。
- 若有不清楚的地方，停下来。指出令你困惑之处，提出问题。

## 2. 简洁优先

**用最少的代码解决问题，不做投机性设计。**

- 不添加超出要求范围的功能。
- 一次性使用的代码不做抽象化。
- 不添加未被明确要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景添加错误处理。
- 如果写了 200 行却可以用 50 行完成，重写它。

问自己："一位资深工程师会认为这过于复杂吗？"如果会，那就简化。

## 3. 精准修改

**只改动必须改动的地方。只清理自己制造的混乱。**

编辑现有代码时：
- 不"优化"与本次任务无关的代码、注释或格式。
- 不重构没有问题的代码。
- 遵循现有的代码风格，即便你有不同偏好。
- 若发现无关的死代码，提及它——但不要删除。

当你的改动产生孤立代码时：
- 删除因**你的改动**而变得无用的导入/变量/函数。
- 不删除已有的死代码，除非被明确要求。

检验标准：每一行改动都应能直接追溯到用户的需求。

## 4. 目标驱动执行

**定义成功标准，循环直至验证通过。**

将任务转化为可验证的目标：
- "添加校验" → "为无效输入编写测试，然后让测试通过"
- "修复 Bug" → "编写能复现 Bug 的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试均通过"

对于多步骤任务，陈述一个简要计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

清晰的成功标准让你能够独立完成循环迭代。模糊的标准（"让它能用"）会导致频繁的反复确认。

---

**这些准则起效的标志：** diff 中不必要的改动减少，因过度复杂而重写的情况减少，澄清性问题在实现前提出而非在出错后才问。
